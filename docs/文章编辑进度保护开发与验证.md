# 文章编辑进度保护开发与验证

## 方案结论

本次采用“本机自动保留 + 路由离开确认 + 浏览器关闭兜底”，不调用服务端接口自动保存草稿。

原因：现有“保存草稿”复用文章创建/更新接口，并会把 `isPublished` 设为 `false`。若自动调用，编辑已发布文章时可能直接下架线上内容；同时接口不支持无标题草稿。后续若要做真正的云端草稿箱，应增加独立 Draft/Revision 资源，与已发布版本隔离。

## 用户行为

- 编辑标题、正文、分类、标签、摘要、封面和发布设置后，900ms 内自动保留到当前浏览器。
- 新建文章、不同文章、不同管理员使用独立草稿键；图片关联用的 `articleHash` 也随草稿恢复。
- 点击侧栏、返回或其他路由时，提示“保留进度并离开 / 继续编辑”。
- 刷新或关闭标签页时使用浏览器原生离开提示；离开前同步写入最后一次输入。
- 再次进入相同编辑页时，提示“恢复进度 / 放弃本机进度”。
- 若服务器文章在草稿产生后已变化，恢复提示会警告用户检查差异，不静默覆盖。
- 手动保存草稿或发布成功后清除本机副本；接口失败时保留本机副本。
- 本机草稿保留 30 天；存储被禁用或容量不足时只警告一次，并阻止误以为已安全保留后离开。

## 存储结构

键：`blog:article-editor-draft:v1:<userId>:<articleId|create>`

值包含：

- `version` / `savedAt`
- `articleHash`
- `editorType`
- `baseFingerprint`（判断服务器版本是否变化）
- 完整编辑表单

## 后端后续契约建议

若要实现跨设备草稿箱，新增独立的 ArticleDraft/Revision 资源，至少支持空标题、版本号或 `If-Match` 冲突控制、草稿与线上发布状态隔离；不要复用当前 Article 更新接口做静默自动保存。
