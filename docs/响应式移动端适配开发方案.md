# 响应式移动端适配开发方案（中间文件）

> 状态：待实施
> 分析对象：`我的博客`（Vue 3 + TypeScript + Vite + Element Plus + UnoCSS + SCSS）
> 决策优先级：**性能 > 代码复用 > 最小改动**
> 目标：在不改变现有产品定位、视觉语言、路由与数据流的前提下，使前台和管理端在手机与平板上可完整使用。

## 1. 结论先行

项目不需要引入新的响应式框架、移动端 UI 库、全局状态或设备检测依赖。现有技术栈已经具备完成适配的全部能力：CSS Grid/Flex、SCSS、Element Plus、UnoCSS，以及已安装的 `@vueuse/core`。

当前状态不是“完全没有响应式”，而是“局部存在但缺少统一闭环”：

- 前台内容区已经具备部分断点规则，三栏布局也会降为单栏。
- `AppNavbar.vue` 在 `<= 768px` 时直接隐藏 `.navbar-menu`，导致搜索、归档、关于、项目、登录/后台入口全部不可用，这是前台最高优先级缺陷。
- `BlogLayout3Col.vue` 在移动端仍渲染左右侧栏，只是排列为单列，造成主内容不是首屏重点，并增加无效 DOM、图片和数据展示成本。
- `AdminLayout.vue` 没有移动端断点，固定侧栏会挤压管理内容；表格、筛选栏、弹窗和编辑器也存在溢出风险。
- 页面断点主要集中在 `768px`，另有后台图表使用 `960px`、三栏使用 `1280px`。可以保留这些既有边界，暂不建立新的断点体系。
- `ParticleMouseBackground.vue` 已在粗指针设备和减少动效偏好下停止初始化，`ReadingProgress` 使用 `requestAnimationFrame`，已有较好的性能基础，应保留。

因此，推荐采用“**先改公共壳层，再做页面兜底**”的实施顺序，而不是重写布局或逐页复制媒体查询。

## 2. 当前架构分析

### 2.1 架构与影响面

```text
router
├─ FrontLayout
│  ├─ ParticleMouseBackground
│  ├─ AppNavbar               ← 移动导航入口缺失
│  ├─ router-view
│  │  └─ BlogLayout3Col       ← Home/Articles/ArticleDetail/Category 共用
│  │     ├─ SidebarLeft
│  │     ├─ 页面主内容
│  │     └─ SidebarRight
│  └─ Footer
└─ AdminLayout                ← 所有后台页面共用，移动侧栏缺失
   ├─ Aside/Menu
   ├─ Header
   └─ router-view
```

公共壳层具有最大的改动收益：调整 `AppNavbar`、`BlogLayout3Col`、`AdminLayout` 三处即可覆盖绝大多数路由。页面层只处理自身确实无法由壳层解决的表格、表单、编辑器和长内容溢出。

### 2.2 已有可复用能力

- `src/styles/index.scss`：三层 Token 的事实来源，已提供 spacing、字体、颜色、圆角、阴影、动效和最大内容宽度。
- `src/components/BlogLayout3Col.vue`：前台四个核心内容页共用，应继续作为响应式布局唯一入口。
- `src/layouts/FrontLayout.vue`：统一控制前台内容边距和 Footer。
- `src/layouts/AdminLayout.vue`：统一控制后台侧栏、Header 和内容容器。
- `@vueuse/core`：如需响应式状态，可使用已有 `useMediaQuery`，不新增依赖。
- Element Plus：已有 Drawer、Table、Dialog 等能力；如使用 Drawer，继续通过现有自动导入机制，不做整库导入。

### 2.3 当前主要风险

| 优先级 | 位置 | 问题 | 用户影响 |
|---|---|---|---|
| P0 | `AppNavbar.vue` | 手机端隐藏整个菜单且没有替代入口 | 核心导航不可达 |
| P0 | `AdminLayout.vue` | 固定侧栏占用窄屏宽度 | 后台主要内容不可用 |
| P1 | `BlogLayout3Col.vue` | 移动端仍渲染并展示两侧栏 | 首屏信息顺序差、额外渲染 |
| P1 | 后台列表页 | 筛选控件固定宽度，表格缺少统一横向容器 | 横向溢出、操作按钮不可见 |
| P1 | 编辑/配置页 | 320/400px 固定区块、双列 Grid | 小屏压缩或溢出 |
| P2 | 内容页 | 长 URL、代码、表格、媒体的兜底不完全一致 | 局部横向滚动 |
| P2 | 交互 | 部分功能依赖 hover，触控语义不完整 | 项目下拉菜单无法稳定操作 |

## 3. 是否需要增加内容

### 3.1 需要增加

1. `AppNavbar.vue` 内增加移动端菜单按钮和轻量菜单面板。
2. `AdminLayout.vue` 内增加移动端抽屉式导航，以及桌面折叠/移动抽屉两个独立状态。
3. `src/styles/index.scss` 增加少量全局“内容安全”规则：图片/视频最大宽度、长文本断行、可滚动表格容器的惯性滚动。
4. 对后台页面增加统一的语义类，例如 `.admin-toolbar`、`.admin-table-scroll`；先在 3 个及以上页面确认重复后再抽为全局规则。
5. 增加移动端验证矩阵和性能预算，本文件第 8、9 节可直接作为验收依据。

### 3.2 不需要增加

- 不新增 Tailwind、Bootstrap、移动端 UI 库或 CSS-in-JS。
- 不新增 Pinia store；菜单展开是布局局部状态。
- 不新增通用 `useResponsive` composable。当前只有少量布局状态，优先 CSS；后台确需状态时直接使用已有 `useMediaQuery`。满足第三次重复后再抽象。
- 不新建 Mobile 版页面或复制组件；同一 DOM 结构通过 CSS/条件展示适配。
- 不修改 API、Store、Router、鉴权和数据结构。
- 不改品牌色、字体体系与 Design Token 公共 API。
- 不用 JavaScript 监听 `resize` 做样式布局；优先 CSS Media Query，避免主线程工作和状态抖动。

## 4. 响应式策略

沿用现有断点，避免扩大改动面：

| 范围 | 断点 | 布局策略 |
|---|---:|---|
| 手机 | `<= 768px` | 单列；移动导航；侧栏不占首屏；工具栏堆叠；弹窗近全宽 |
| 平板/小桌面 | `769px - 1279px` | 前台双栏；后台可折叠侧栏；内容保持弹性 |
| 大桌面 | `>= 1280px` | 保留现有前台三栏 |
| 后台图表紧凑态 | `<= 960px` | 沿用现有 Dashboard 断点，双列图表降单列 |

原则：使用 `min-width: 0`、`max-width: 100%`、`minmax(0, 1fr)` 和内容换行解决弹性问题；只在布局确实改变时使用断点，避免为每个设备尺寸写特例。

## 5. 实施方案（按优先级）

### Phase A：P0 公共壳层可用性

#### A1. 前台移动导航 — `src/components/AppNavbar.vue`

- 保留桌面 `.navbar-menu`，手机端隐藏桌面菜单。
- 增加语义化 `<button>` 作为菜单触发器，具备 `aria-expanded`、`aria-controls`、可见 focus 状态和至少 44×44px 触控区域。
- 移动面板复用现有 `navItems`、`projects`、登录状态和搜索方法，不复制数据源。
- 项目入口在移动端改为可点击分组/列表，不能依赖 `mouseenter`。
- 路由变化、按 Escape、点击遮罩后关闭菜单；菜单打开时控制页面滚动，关闭/卸载时恢复。
- 搜索弹层增加 `role="dialog"`、`aria-modal="true"`、输入框 label，并在打开后聚焦。
- 不引入 Drawer 也可以完成；优先原生 fixed panel + CSS transition，产物更小。若团队更重视统一交互，可使用已安装的 Element Plus Drawer，但需对比构建体积后决定。

性能要求：菜单关闭时使用 `v-if` 不保留面板 DOM；动效仅使用 `transform` 和 `opacity`；不得使用持续监听尺寸的 JS。

#### A2. 后台移动导航 — `src/layouts/AdminLayout.vue`

- 桌面端保留现有 `el-aside` 和折叠行为。
- 手机端不渲染桌面侧栏，Header 显示菜单按钮，通过 `el-drawer` 承载同一份菜单结构。
- 为避免复制菜单模板，先在当前文件内以菜单配置数组驱动桌面/移动两个 `el-menu`；只有出现第三个消费者时再抽独立组件。
- 使用已有 `useMediaQuery('(max-width: 768px)')` 区分布局状态，不新增依赖。
- 路由变化后自动关闭 Drawer。
- 移动端隐藏或压缩 breadcrumb、用户名，只保留头像与主题按钮；Header 交互区不得溢出。
- `.admin-main` 在手机端减少 padding，并设置 `min-width: 0`、`overflow-x: hidden`；具体表格在自身容器横向滚动，不让整页滚动。

性能要求：移动 Drawer 使用 `v-if="isMobile"`，桌面 Aside 使用 `v-else`，避免同时保留两套较重菜单 DOM。

#### A3. 前台内容顺序 — `src/components/BlogLayout3Col.vue`

- 手机端主内容必须排第一。
- 左侧作者信息如对移动访客仍有价值，改为主内容后的紧凑区；右侧默认 widgets 在手机端隐藏。
- 使用 CSS 控制，不增加设备状态或重复组件。
- `769px - 1279px` 保持现有双栏，`>=1280px` 保持三栏。

建议手机顺序：`主内容 → 作者/站点信息 → 可选辅助内容`。如果移动端右侧内容没有明确转化价值，直接 `display: none`，不要为了“信息完整”牺牲首屏性能。

### Phase B：P1 页面级溢出治理

#### B1. 后台列表页

目标文件：

- `src/views/admin/ArticleList.vue`
- `src/views/admin/CategoryList.vue`
- `src/views/admin/TagList.vue`
- `src/views/admin/VisitLogs.vue`

实施方式：

- 筛选栏统一允许 `flex-wrap`，手机端控件宽度 `100%`，移除模板中的固定 inline width，改用语义类。
- 表格外包一层横向滚动容器；保留关键列，次要列可在 `<=768px` 隐藏。
- 操作按钮避免全部文字横排，可收敛为下拉菜单，但不改变权限和业务方法。
- 分页在手机端采用简化布局或 Element Plus small 模式。

#### B2. 后台表单、编辑器与配置页

目标文件：

- `src/views/admin/Settings.vue`
- `src/views/admin/ArticleEditor.vue`
- `src/views/admin/PageConfigEditor.vue`

实施方式：

- 双列 Grid 在手机端降为单列。
- 固定 `320px/400px` 区块改为 `width: min(100%, 320px/400px)` 或手机端 `width: 100%`。
- Element Plus Form 的 label 在窄屏切换顶部布局；弹窗宽度使用 `min(90vw, 既有宽度)`。
- 编辑器工具栏允许内部横向滚动，不让整个页面溢出。
- 二维码、OTP 输入、备用码网格保持可触控，输入单元不能小于可用宽度。

#### B3. 前台内容兜底

目标文件：`ArticleDetail.vue`、`Articles.vue`、`CategoryArticles.vue`、`Home.vue`、`Projects.vue`、`ProjectDetail.vue`、`About.vue`。

- 长标题和 URL 使用 `overflow-wrap: anywhere`。
- 代码块、表格只在自身容器滚动。
- 图片、视频、canvas 使用 `max-width: 100%; height: auto`。
- 卡片 padding 在手机端降一级，不改变视觉 token。
- hover 动效通过 `@media (hover: hover) and (pointer: fine)` 限定，避免触控端粘滞 hover。
- 触控目标最小 44×44px；纯装饰动画继续遵守 `prefers-reduced-motion`。

### Phase C：P2 清理与复用收敛

- 当 `.admin-toolbar`、`.admin-table-scroll`、移动表单规则在至少 3 处重复后，移入 `src/styles/index.scss` 或创建后台布局级样式；不提前抽象。
- 检查所有业务组件是否仅消费 Semantic Token；不要在本次适配中引入新 raw hex。
- 把已验证的响应式模式补充到项目知识库时，应记录“为什么抽象”和消费者清单，而不只记录代码片段。

## 7. 性能设计（第一优先级）

### 7.1 约束

- 不因适配新增第三方运行时依赖。
- CSS 完成布局，JS 只管理菜单开关、焦点和可访问性状态。
- 所有滚动监听保持 `passive`，视觉更新使用 `requestAnimationFrame`；禁止新增高频 `resize` 回调。
- 移动端不初始化粒子背景；现有实现已满足，保持不变。
- 移动端隐藏的重组件优先条件渲染，而不是只做视觉隐藏；纯静态轻量侧栏可用 CSS 隐藏。
- 动画只改变 `transform`/`opacity`，避免 `width`、`left`、`box-shadow` 的连续动画导致布局或高额绘制。
- 图片继续使用现有懒加载指令，并明确 `width/height` 或 `aspect-ratio` 以减少 CLS。
- 不通过 User-Agent 判断设备，使用能力查询：`pointer`、`hover`、`prefers-reduced-motion`。

### 7.2 性能预算

以生产构建和中端 Android 手机为基线：

| 指标 | 目标 |
|---|---:|
| 本次新增 gzip JS | `<= 5 KB`（理想为 0 外部依赖） |
| 移动菜单打开响应 | `<= 100 ms` |
| Lighthouse Mobile Performance | `>= 90`（无后端异常时） |
| LCP | `<= 2.5 s` |
| INP | `<= 200 ms` |
| CLS | `<= 0.1` |
| 页面横向滚动 | 0（代码块/表格自身滚动除外） |

构建前后保存 `dist/assets` 文件大小对比。若使用 Element Plus Drawer 导致公共 chunk 明显增长，应改为原生轻量面板。

## 8. 验证矩阵

### 8.1 视口

- 320×568：最小支持宽度，检查无横向页面滚动。
- 360×800：常见 Android。
- 390×844：常见 iPhone。
- 768×1024：平板边界。
- 1024×768：小桌面/横屏平板。
- 1280×800、1440×900：确认桌面布局无回归。

### 8.2 路由

- 前台：`/`、`/articles`、`/article/:id`、`/category/:id`、`/about`、`/projects`、`/project/:id`。
- 独立页：`/login`、404、无权限页。
- 后台：Dashboard、文章列表/编辑、分类、标签、设置、页面配置、访问日志。

### 8.3 必测行为

- 键盘和触控均可打开/关闭菜单；Escape 可关闭；焦点状态可见。
- 打开移动菜单后背景不可误滚，关闭后滚动恢复。
- 路由跳转后菜单自动关闭。
- 320px 下没有整页横向滚动。
- 表格/代码块横向滚动时不带动整个页面。
- 深色/浅色主题均通过。
- `prefers-reduced-motion: reduce` 下没有非必要位移动画。
- 粗指针设备不运行粒子 Canvas。
- 旋转屏幕、浏览器返回、刷新深层路由后状态正常。

## 9. 实施与验收命令

每个 Phase 独立提交和验证，便于定位回归：

```bash
npx vue-tsc --noEmit
npm run lint
npm run build
```

浏览器验证应同时检查 Console、Network 和 Performance。后台接口未启动时，可以验证静态布局，但不得把接口错误误判为响应式缺陷。

## 10. 推荐执行顺序与停止条件

1. 完成 `AppNavbar.vue`、`AdminLayout.vue`、`BlogLayout3Col.vue`。
2. 通过全部核心路由的 320/390/768/1280 宽度检查。
3. 修后台列表页真实溢出。
4. 修后台编辑/设置页真实溢出。
5. 修前台剩余页面真实溢出。
6. 只有出现 3 次以上重复时才抽取公共响应式类。
7. 对比构建产物与 Lighthouse，性能预算不通过则停止扩展并回退高成本实现。

完成标准不是“所有页面都有 media query”，而是：核心功能在 320px 起完整可达、无整页横向溢出、桌面无回归、没有新增运行时依赖，并达到第 7 节性能预算。

## 11. 本方案明确排除的范围

- 不重新设计品牌或信息架构。
- 不重构 API、Store、Router 或权限系统。
- 不同时改造成 PWA、SSR 或独立移动站。
- 不为所有页面建立新的组件层级。
- 不为尚未出现的复用场景预建抽象。

这些事项可单独立项，不能与本次响应式适配捆绑，否则会破坏最小改动原则并增加回归面。
